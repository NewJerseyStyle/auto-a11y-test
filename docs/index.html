üìã Prerequisites

Required Secrets

You'll need to configure these secrets in your GitHub repository settings:

  1. AI_API_KEY: Your OpenAI or Groq API key
  2. AI_PROVIDER: Either "openai" or "groq"
‚ö†Ô∏è Important: Never commit API keys directly in your code. Always use GitHub Secrets!

Setting Up Secrets

  1. 1 Go to your repository on GitHub
  2. 2 Click on Settings ‚Üí Secrets and variables ‚Üí Actions
  3. 3 Click "New repository secret"
  4. 4 Add your AI_API_KEY and AI_PROVIDER

üîß Installation & Setup

Method 1: Using the GitHub Action Directly

Create a workflow file in your repository:

.github/workflows/accessibility-test.yml
name: AI Accessibility Testing

on:
  pull_request:
  push:
    branches: [main]
  workflow_dispatch:

jobs:
  accessibility-test:
    runs-on: ubuntu-latest
    
    steps:
    - name: Checkout code
      uses: actions/checkout@v4
    
    - name: Run AI Screen Reader Test
      uses: NewJerseyStyle/AI-Website-ScreenReader-Navigation-Tester@main
      with:
        url: 'https://your-website.com'
        goals: |
          Navigate to the contact page
          Find the phone number
          Locate the business hours
        ai-provider: ${{ secrets.AI_PROVIDER }}
        ai-api-key: ${{ secrets.AI_API_KEY }}
        max-steps: '15'
        
    - name: Upload Test Results
      if: always()
      uses: actions/upload-artifact@v4
      with:
        name: accessibility-report
        path: accessibility-report/

Method 2: Testing Locally

Clone the repository and run tests locally:

git clone https://github.com/NewJerseyStyle/AI-Website-ScreenReader-Navigation-Tester.git
cd AI-Website-ScreenReader-Navigation-Tester
npm install
npm run test:local

ü§ù Best Practices

Writing Effective Goals

  • ‚úÖ Be specific but not prescriptive: "Find contact information" not "Click the Contact link in the header"
  • ‚úÖ Focus on user outcomes: "Purchase a product" not "Click add to cart button"
  • ‚úÖ Test critical user journeys
  • ‚úÖ Include error scenarios: "Find error message when form is invalid"

Accessibility Guidelines

  • Ensure all interactive elements have proper ARIA labels
  • Use semantic HTML (headers, nav, main, footer)
  • Provide keyboard navigation support
  • Include skip links for navigation
  • Test with multiple screen reader combinations

CI/CD Integration Tips

  • Run tests on pull requests to catch issues early
  • Use matrix testing for different browsers/viewports
  • Store test artifacts for debugging
  • Set up notifications for test failures
  • Consider running nightly tests for comprehensive coverage

üêõ Troubleshooting

Common Issues and Solutions

Issue: "API key not valid"
Solution: Verify your API key is correctly set in GitHub Secrets and has sufficient credits/permissions.
Issue: "Screen reader not starting"
Solution: The action only works on Linux/macOS runners. Ensure you're using ubuntu-latest or macos-latest.
Issue: "Navigation timeout"
Solution: Increase the max-steps parameter or simplify your goals.
Issue: "Cannot find element"
Solution: Check that your site has proper ARIA labels and semantic HTML structure.
